# ТЗ на мини-приложение

---

## 1. Общее описание

Необходимо разработать мобильное приложение, которое отображает данные из открытого API Беларусбанка. Приложение поддерживает регистрацию и авторизацию пользователя, хранит состояние сессии между запусками, а также позволяет переключать тему оформления.

---

## 2. Навигационная схема

```
Запуск приложения
│
├── Первый запуск → Онбординг → Экран выбора (Войти / Зарегистрироваться)
│
├── Повторный запуск, пользователь авторизован → Главный экран (TabBar)
│
└── Повторный запуск, пользователь вышел из профиля → Экран входа
```

---

## 3. Экраны и функциональные требования

### 3.1 Онбординг

- Отображается **только при первом запуске** приложения.
- Состоит из 2–3 слайдов с кратким описанием возможностей приложения.
- На последнем слайде — кнопка **«Начать»** или **«Далее»**.
- После завершения онбординга — переход на экран выбора входа/регистрации.
- При повторном запуске приложения онбординг **не отображается**.

> Флаг `onboardingCompleted` сохраняется локально (например, `UserDefaults` / `SharedPreferences`).

---

### 3.2 Экран выбора: Войти / Зарегистрироваться

- Два варианта действия:
  - **Войти** — переход на экран входа (3.3)
  - **Зарегистрироваться** — переход на экран регистрации (3.4)

---

### 3.3 Экран входа (Login)

**Поля:**
- Логин (текстовое поле)
- Пароль (поле с маскировкой ввода)

**Кнопки:**
- **Войти** — валидация, проверка введённых данных с сохранёнными учётными данными. При успехе — переход на главный экран (TabBar).
- **Зарегистрироваться** — переход на экран регистрации (3.4).

**Валидация:**
- Поля не должны быть пустыми.
- При ошибке — отображение сообщения об ошибке.

---

### 3.4 Экран регистрации (Register)

**Обязательные поля:**
- Логин
- Пароль
- Подтверждение пароля

**Дополнительные поля (по желанию):**
- Имя
- Email
- Телефон

**Кнопки:**
- **Зарегистрироваться** — сохранение данных, переход на экран входа (3.3).
- **Назад** — возврат на предыдущий экран.

**Валидация:**
- Все обязательные поля должны быть заполнены.
- Пароль и подтверждение пароля должны совпадать.
- Логин должен быть уникальным (среди ранее зарегистрированных).
- При ошибке — отображение сообщения.

> Данные пользователя хранятся локально (например, `UserDefaults` / `CoreData` / `Realm`).

> ### 🆕 НОВЫЕ ТРЕБОВАНИЯ 35-го урока
>
> Пароль пользователя должен храниться **в Keychain**, а не в `UserDefaults`. Это более безопасный способ хранения чувствительных данных на iOS.
>
> Реализовать можно двумя способами:
>
> **Способ 1 — нативный**, через фреймворк `Security` (входит в iOS SDK, дополнительная установка не нужна)
>
> **Способ 2 — через стороннюю библиотеку** (SPM), например:
> - [KeychainAccess](https://github.com/kishikawakatsuki/KeychainAccess)
>
> Подключение через **Swift Package Manager**: `File → Add Package Dependencies → вставить URL репозитория`.

---

### 3.5 Главный экран (TabBar)

TabBar содержит **две вкладки:**

| Иконка | Название | Описание |
|--------|----------|----------|
| 🏠 | Главная | Данные из API Беларусбанка |
| ⚙️ | Настройки | Тема и выход из профиля |

---

### 3.6 Вкладка «Главная»

Отображает данные из открытого API Беларусбанка:  
📎 [https://belarusbank.by/o-banke/for-developers/informatsionnye-api/](https://belarusbank.by/o-banke/for-developers/informatsionnye-api/)

**Студент реализует на выбор два из следующих разделов:**

| Раздел | Описание |
|--------|----------|
| Курсы валют | Текущие курсы обмена валют |
| Депозиты | Список доступных депозитных программ |
| Новости | Последние новости банка |
| Отделения/банкоматы | Список точек обслуживания | (Это нам нужно будет в будущем)

**Требования к отображению:**
- Данные загружаются при открытии экрана.
- На время загрузки отображается индикатор активности (spinner / skeleton).
- При ошибке сети — отображается сообщение об ошибке и кнопка повтора.
- Данные отображаются в виде списка (таблица, карточки или список. В целом дизайн не так важен).

---

### 3.7 Вкладка «Настройки»

**Элементы экрана:**

1. **Переключатель темы** — светлая / тёмная / системная  
   - Выбранная тема применяется немедленно ко всему приложению.  
   - Тема **сохраняется между запусками** приложения.  
   - При следующем запуске приложение стартует с сохранённой темой.
   - Решения этой задачи вы сможете найти в интернете. Важно в поиске укзаать что вам нужно переключать темы с использованием UIKit

2. **🆕 НОВЫЕ ТРЕБОВАНИЯ 35-го урока — Переключатель уведомлений (Toggle «Получать уведомления»)**
   - Пользователь может включить или выключить ежедневные уведомления.
   - Состояние переключателя **сохраняется** между запусками (`UserDefaults`).

   **Поведение при включении:**
   1. Приложение запрашивает разрешение на отправку уведомлений через `UNUserNotificationCenter.requestAuthorization`.
   2. Если пользователь **разрешил** — планируется ежедневное локальное уведомление в фиксированное время (например, в 12:00).
   3. Если пользователь **отклонил** разрешение — переключатель возвращается в положение «выкл», пользователю показывается сообщение с предложением разрешить уведомления в настройках устройства.

   **Поведение при выключении:**
   - Все запланированные уведомления отменяются.

   **Содержание уведомлений** — шуточные, банковской тематики. Каждый день случайно выбирается одно из сообщений

3. **Кнопка «Выйти из профиля»**  
   - При нажатии — очищается сессия пользователя (флаг авторизации сбрасывается).  
   - Пользователь перенаправляется на экран входа (3.3).  
   - Онбординг при этом **не показывается** повторно.

## 4. Логика запуска приложения

При каждом запуске приложение проверяет сохранённые флаги и выбирает начальный экран:

```
┌─────────────────────────────────────────────────────────────────┐
│                     Запуск приложения                           │
└────────────────────────────┬────────────────────────────────────┘
                             │
             isOnboarding == true?
                    ┌────────┴────────┐
                   ДА               НЕТ
                    │                │
              Онбординг     isUserLoggedIn == true?
                             ┌───────┴───────┐
                            ДА             НЕТ
                             │               │
                         TabBar        Экран входа
```

---

## 5. Хранение данных

| Данные | Способ хранения | Описание |
|--------|-----------------|----------|
| Флаг онбординга | UserDefaults | `isOnboarding: Bool` |
| Флаг авторизации | UserDefaults | `isUserLoggedIn: Bool` |
| Логин пользователя | UserDefaults / CoreData / Realm | Строка |
| **🆕 Пароль пользователя** | **Keychain** | Секретные данные — только Keychain |
| Выбранная тема | UserDefaults | `selectedTheme: String` |
| **🆕 Флаг уведомлений** | **UserDefaults** | `notificationsEnabled: Bool` |

---

## 6. Требования к внешнему виду

- Приложение поддерживает **светлую и тёмную тему**.
- Интерфейс должен быть **аккуратным и читаемым** — строгих дизайн-требований нет.
- Шрифты, цвета и отступы — на усмотрение.
- Допускается использование системных компонентов платформы.

---

## 7. Технические требования

- Сетевые запросы через `URLSession` / `Async/await` / `GCD`.
- Данные из API парсятся из **JSON**.
- Хранение настроек: `UserDefaults` / `CoreData` / `Realm`
- Сторонние библиотеки — разрешены
